# Temu商品发布自动化项目实施方案

**编制日期**：2025年10月25日  
**预计工期**：1个月



经过对比Playwright纯代码、n8n工作流、影刀RPA三种方案，最终确定采用**影刀RPA + Python混合架构**：

### 选型理由
- 影刀处理浏览器交互，开发效率高，有录制功能可以快速出原型
- Python处理数据逻辑，灵活性强，便于后期扩展
- 两者结合兼顾开发速度和系统可维护性
- 影刀个人版免费，如果不够用再升级专业版

### 架构设计
```
选品表(Excel) → Python读取处理 → 生成任务数据(JSON) 
                                        ↓
影刀执行浏览器操作 ← Python调度 ← 任务队列
         ↓
     结果反馈(JSON) → Python记录日志 → 数据库
```



## 实施计划

### 第一周（7个工作日）

**目标**：完成基础框架和前半段流程

#### Day 1-2：环境准备
- 安装影刀客户端（桌面版）
- 搭建Python开发环境
- 确定数据交互方式（JSON文件）
- 初始化Git仓库

#### Day 3：Python数据处理层
- 读取选品表Excel
- 实现AI标题生成（优先用Temu自带功能，不行就调API）
- 价格计算逻辑（成本×2.5×3）
- 预计遇到的问题：选品表格式可能不统一，需要多测几个样本

#### Day 4：影刀登录流程
- 录制Temu后台登录操作
- Cookie保存（减少重复登录）
- 验证码处理：先手动介入，后续看情况优化（打算用一个 qwen3-vl解决）

#### Day 5：影刀搜索采集
- 录制站内搜索和链接采集
- 一次采集5个同款商品
- 输出链接列表到JSON
- 难点：商品筛选逻辑需要反复调试

#### Day 6-7：首次编辑流程
- 修改标题、核对类目
- 图片处理（MVP阶段简化：人工确认+脚本替换）
- 认领4次操作（5条变20条）
- 这块操作步骤多，预计需要反复测试

**Week1验收**：能够从选品表自动完成搜索采集和首次编辑

---

### 第二周（7个工作日）

**目标**：完成核心批量编辑和发布功能

#### Day 8-11：批量编辑18步（核心且最耗时）
- 分批录制18个编辑步骤
- 重点：类目属性这块比较复杂，可能需要2-3天
- 参数化处理（从JSON读配置）
- 每步增加错误处理
- 注意：SOP文档中的步骤可能和实际操作有偏差，需要边做边调整

批量编辑的18个步骤：
1. 修改标题、英语标题（空格键触发）
2. 修改类目属性（参考采集链接，最复杂）
3. 修改主货号（跳过）
4. 修改外包装（长方体/硬包装/图片上传）
5. 修改产地（浙江）
6. 修改定制品（跳过）
7. 修改敏感属性（跳过）
8. 修改重量（5000-9999G随机）
9. 修改尺寸（50-99cm，长>宽>高）
10. 修改平台SKU（自定义编码）
11. 修改SKU分类（组合装500件）
12. 尺码表（跳过）
13. 修改建议售价（成本×10）
14. 修改包装清单（跳过）
15. 轮播图（跳过）
16. 颜色图/预览图（跳过）
17. 修改产品说明书（上传文件）
18. 保存并验证

#### Day 12-13：批量发布
- 选择店铺、设置供货价
- 批量发布操作（点击2次，共40条）
- 发布结果抓取和统计
- 失败原因分析

#### Day 14：Python流程编排
- 主程序串联所有流程
- 调用影刀的方式（命令行或API）
- 失败重试3次
- 断点续传（做到什么程度看时间）

### 第三周前半段（3-4天）

**目标**：测试和交付

#### Day 15-16：完整测试
- 跑通至少3个不同类型的产品
- 记录错误率（目标控制在20%以内）
- 发现问题及时修复
- 性能测试：看单批次要多久

#### Day 17：文档和交付
- 使用说明文档
- 常见问题汇总
- 代码整理
- 给下一个接手的人留点注释

## 风险与应对

### 技术风险

#### 1. 网站反爬虫机制
- **风险等级**：中
- **应对**：降低操作频率，每步加1-3秒随机延迟；使用真实浏览器环境；减少登录次数
- **如果被封**：换账号，或者联系平台看能否加白名单

#### 2. 页面元素变化
- **风险等级**：中高
- **应对**：影刀的元素定位支持多种方式，配置备用方案；关键元素变化后需要及时更新脚本
- **建议**：每周检查一次流程是否正常

#### 3. 图片验证困难
- **风险等级**：低
- **应对**：MVP阶段降级为人工辅助验证；后续有精力再做图像识别




## 后续优化方向

如果MVP版本跑通，后续可以考虑：

1. 图片自动验证（接入图像识别）
2. 多账号并发处理（提升效率）
3. Web管理界面（方便运营人员使用）
4. 更详细的数据统计和报表
5. 移动端监控（随时查看执行状态）




**备注**：本方案为MVP（最小可行产品）版本，重点解决核心流程自动化问题。后续功能优化根据实际使用情况和反馈逐步迭代。
